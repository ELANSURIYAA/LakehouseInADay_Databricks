DC Health Meter - Conceptual Analytics Data Model (Process Tables)

The conceptual analytics model for DC Health Meter includes core dimensions (entities) and core fact tables supporting:
Operations, Inventory, Headcount, Picks, Kronos, and Exceptions.

Below tables list suggested business columns, data types, constraints, and domain values.
Note: This is a conceptual-to-logical bridge (minimal attributes). Physical attributes and platform-specific types will be finalized in the logical model.

Date_Key                | Date                                                              | INT              | Foreign Key (Dim_Date), Not Null  | N/A

Activity_ID             | Activity                                                          | INT              | Foreign Key (Dim_Activity), Not Null | N/A
Partner_ID              | Partner / operator                                                | INT              | Foreign Key (Dim_Partner), Not Null | N/A

FACT: Fact_Exceptions
Grain: One record per Exception Event (Exception Event ID)
-------------------------------------------------------------------------------------------------------------------------------------------------------------------------

ExceptionType_ID        | Exception type                                                      | INT              | Foreign Key (Dim_ExceptionType), Not Null | N/A

Item_ID                 | Item impacted (if applicable)                                       | INT              | Foreign Key, Nullable             | N/A

* Prevent duplicates on Exception_Event_ID.

-------------------------------------------------------------------------------------------------------------------------------------------------------------------------
GLOBAL DATA QUALITY / CONSTRAINT GUIDELINES (applies to all tables)
-------------------------------------------------------------------------------------------------------------------------------------------------------------------------
* Prevent duplicate entries for unique fields (e.g., DC_Name, Partner_Name, Item_ID, Calendar_Date).
* Enforce referential integrity for all foreign keys to their respective dimensions.
* Standardize domain values using reference dimensions (Activity, Equipment, PartnerType, ExceptionType).
* Where employee data is present, apply masking/hashing and role-based access controls (PII governance).
